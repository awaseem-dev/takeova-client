# MINE Native App

Native iOS + Android shell that wraps the MINE web dashboards and adds native capabilities the web platform can't access — including **Tap to Pay on iPhone**.

## What's in this folder

```
app/
├── package.json                  # Capacitor + plugin deps
├── capacitor.config.ts           # Native config — points WebView at MINE backend
├── src/
│   ├── index.html                # Native boot splash (1s, then loads MINE web)
│   └── main.js                   # Bridges Capacitor APIs to MINE web via window.MineNative
├── plugins/
│   └── stripe-terminal/          # Custom Capacitor plugin for Tap to Pay
│       ├── package.json
│       ├── MineStripeTerminal.podspec
│       ├── src/
│       │   ├── index.ts          # TypeScript interface
│       │   └── index.js          # JS shim
│       └── ios/
│           └── Sources/StripeTerminalPlugin/StripeTerminalPlugin.swift
└── README.md                     # This file
```

`ios/` and `android/` folders **don't exist yet** — they're generated by Capacitor when you run `npx cap add ios` and `npx cap add android`.

## How MINE web detects the native app

The MINE web dashboards check `window.MineNative` to know they're running inside the app:

```js
if (window.MineNative) {
  // Use native capabilities
  const photo = await MineNative.takePhoto();
} else {
  // Fall back to web equivalent
  document.getElementById('fileInput').click();
}
```

All native bridges go through `window.MineNative`. See `src/main.js` for the full API.

## Everything MINE can do — and how the native app handles it

| MINE feature | Web fallback | Native enhancement |
|---|---|---|
| Site Builder (AI) | Works in WebView | + camera for brand photos via `MineNative.takePhoto()` |
| Image upload | `<input type=file>` | + native camera/gallery picker |
| QR code scanning | None | + on-device ML Kit scanner via `MineNative.scanQR()` |
| Push notifications | Browser web push (limited on iOS) | + native APNs/FCM with badges, sounds |
| File downloads (PDFs, contracts) | Browser download | + saves to Documents folder, can share |
| Share invoice link | Manual copy/paste | + native iOS/Android share sheet |
| Apple Pay / Google Pay (customer-side) | Already auto-shown by Stripe Checkout | + smoother Universal Link return |
| **Tap to Pay on iPhone** | **Not possible — web can't access NFC** | **Native plugin uses Stripe Terminal SDK** |
| QR code at point-of-sale | SVG image in modal | + same, plus haptic feedback on success |
| Mailchimp import | API key paste | Same |
| Shopify import | API key paste | Same |
| AI Employees | All work | Same |
| Invoices / Orders / Contacts | All work | + biometric unlock optional |
| Geolocation (AI Receptionist) | Browser geo | + more accurate native GPS |
| Camera (product photos) | Browser API | + faster, native UX |
| Keyboard control | Limited | + dismiss / resize handling |
| Offline | Doesn't work | + cached login + offline queue |
| Deep links (mine://invoice/123) | Doesn't work | + opens specific screen |

## Setup (first time)

### 1. Prerequisites

- **macOS** (required for iOS build — Linux/Windows works for Android only)
- **Xcode 15+** (free from Mac App Store)
- **Node.js 20+**
- **CocoaPods** (`sudo gem install cocoapods`)
- **Apple Developer Program** account ($99/year — required to install on real devices and submit to App Store)
- **Android Studio** (optional, for Android build)

### 2. Install dependencies

```bash
cd app
npm install
```

### 3. Generate native project folders

```bash
npx cap add ios
npx cap add android   # optional
```

This creates `ios/` and `android/` directories with the actual Xcode project and Android Studio project. **Commit these to git after generating.**

### 4. Configure iOS for Tap to Pay

Open `ios/App/App.xcworkspace` in Xcode, then:

**a. Set the Bundle Identifier**
- Targets > App > Signing & Capabilities > Bundle Identifier: `app.mine.dashboard` (or whatever you set in `capacitor.config.ts`)

**b. Add capabilities** (Signing & Capabilities tab → + Capability):
- ✅ **Push Notifications**
- ✅ **Background Modes** → Remote notifications, Background fetch

**c. Request the Tap to Pay entitlement**
- Sign in to https://developer.apple.com/account
- Identifiers → click your bundle ID → Capabilities
- Check "Tap to Pay on iPhone" → Save
- Wait for Apple approval (2-4 weeks typical, can be longer)
- After approval: Xcode → Targets > App > Signing & Capabilities → + Capability → Tap to Pay on iPhone

**d. Add Stripe Terminal pod**

Edit `ios/App/Podfile`, add inside the target block:
```ruby
pod 'StripeTerminal', '~> 3.0'
```

Then run:
```bash
cd ios/App
pod install
```

**e. Add the Stripe Terminal plugin to Podfile**

Your local plugin gets auto-linked by Capacitor when registered. Verify it's in the Podfile.

**f. Set minimum iOS version**
- Targets > App > General > Minimum Deployments → iOS 16.4

### 5. Set the backend URL

Edit `capacitor.config.ts`:

```typescript
server: {
  url: 'https://YOUR-MINE-BACKEND.com',  // change this
}
```

Then sync:
```bash
npx cap sync ios
```

### 6. Run on simulator

```bash
npx cap open ios
```

Then click ▶ in Xcode. Tap to Pay will NOT work in simulator — only on a real device.

### 7. Run on real device

In Xcode:
- Connect iPhone via USB
- Trust the developer cert on the phone (Settings > General > VPN & Device Management)
- Select your iPhone in the device selector at the top of Xcode
- Click ▶

## Tap to Pay flow (when everything is set up)

```
MINE web user opens Quick Pay
    ↓
Enters customer details + amount
    ↓
Picks "Tap to Pay" option (only visible inside the app)
    ↓
JS calls: window.MineNative.tapToPayCharge({amount, invoice_id})
    ↓
Native plugin (Swift) takes over:
  1. Fetches Stripe connection token from MINE backend
  2. Initializes Stripe Terminal SDK with Tap to Pay reader
  3. Backend creates PaymentIntent with payment_method_types=['card_present']
  4. Swift presents Apple's payment sheet on phone back
  5. Customer taps their contactless card OR Apple Pay phone
  6. Stripe captures payment, returns success
  7. Backend webhook marks the invoice paid
    ↓
JS receives result: { status:'succeeded', last4:'4242', brand:'visa' }
    ↓
MINE web shows success screen, contact saved, invoice updated
```

## Submitting to the App Store

After Tap to Pay testing works on a real device:

1. **App Store Connect** → My Apps → + New App
2. Fill in metadata (name, screenshots, description)
3. Privacy policy URL — REQUIRED. Mention NFC, payment processing.
4. In Xcode: Product → Archive → Distribute App → App Store Connect
5. Wait for App Review (typically 24-72 hours)
6. App goes live

**Common rejection reasons for payment apps:**
- Missing privacy policy
- Tap to Pay entitlement not yet approved
- Incomplete screenshots
- "Designed for iPad" not checked but app works on iPad

## Updating the app after release

**You only need to ship an app update when:**
- You add new native bridges in `main.js`
- You change Capacitor plugins
- Apple/Google require a SDK update

**You DON'T need a new app version when:**
- You update MINE web dashboards — they live on your backend, the WebView refreshes them
- You add new MINE features (AI Employees, integrations, etc.)
- You change MINE UI/styling

This is the magic of the WebView wrapper pattern: 99% of updates are server-side.

## What's NOT built (yet, by design)

- Android Tap to Pay (Stripe Terminal supports it via different setup — easy to add once iOS works)
- Apple Wallet pass for invoice receipts
- Biometric login (FaceID/TouchID) — easy to add via `@capacitor/biometric-auth`
- Offline mode — currently online-only
- Native settings screens — currently uses web dashboards

These are all "v2 polish" items, not v1 blockers.

## Troubleshooting

**"Tap to Pay isn't available on this device"**
- Check iPhone model: must be XS or newer
- Check iOS version: must be 16.4+
- Verify entitlement is granted (Apple Developer Portal)
- App must be signed with a provisioning profile that includes the entitlement

**"Connection token request failed"**
- Backend URL in `capacitor.config.ts` must be reachable from the device
- The user must be logged in (`localStorage.mine_token` must be set)
- Backend must have `STRIPE_SECRET_KEY` set

**"App rejected by Apple"**
- Email the rejection notice — most are easy to fix
- Common: privacy policy missing, mention of Tap to Pay requires Apple's explicit naming
